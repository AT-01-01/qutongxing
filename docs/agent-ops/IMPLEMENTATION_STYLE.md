# 实现风格规范（最简最优 + 详细注释）

## 1. 核心原则
- 最小改动优先：用最少文件、最短链路完成需求目标。
- 复杂度受控：避免引入当前需求不需要的新抽象、新框架、新配置。
- 必要完整：需求直接相关的校验、错误处理、测试必须具备。
- 非必要不做：禁止顺手重构、禁止扩展无关能力。

## 2. 方案选择规则
- 若有多个可行方案，优先级为：
  1. 改动更小
  2. 可读性更好
  3. 与现有代码风格更一致
  4. 回归成本更低
- 任何“更通用但更复杂”的方案，除非当前需求必须，否则不采用。

## 3. 注释强制要求
- 关键逻辑必须有注释，尤其是：
  - 业务分支判断
  - 边界条件处理
  - 异常处理与降级策略
  - 跨模块依赖与契约假设
- 注释必须解释“为什么这样做”，不是重复“做了什么”。
- 注释粒度可适当详细，以降低审视和维护成本。

## 4. 注释实践建议
- 在复杂分支前写块注释，说明输入条件、预期输出、失败处理。
- 在重试、鉴权、状态流转等高风险逻辑处写清规则来源。
- 在临时兼容方案处写明触发条件和后续清理条件。

## 5. 禁止项
- 禁止为了“看起来优雅”而增加层级与抽象。
- 禁止将 TODO 留在关键路径且无明确 owner/时限。
- 禁止缺少注释的复杂逻辑直接交付。

## 6. 交付检查
- 是否可以在 5 分钟内讲清改动目的、边界和风险？
- 是否能通过注释快速定位关键逻辑与失败路径？
- 是否删掉了所有与需求无关的额外改动？
